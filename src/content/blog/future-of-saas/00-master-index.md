## Core Question

In a world where AI lets anyone build personal and agentic tools, when and why do SaaS companies still matter?

The answer running through this packet is:

- AI is rapidly commoditizing interface and orchestration.
- SaaS remains durable where it owns shared data, invariants, integrations, governance, and operational reliability.
- The winning strategy is to design for that split instead of fighting it.

## Sharpened Thesis

The old question was: can we build this software at all?

The new question is: which parts of this stack should we own as agents and internal tools, and which parts should still be rented as SaaS backbones?

A useful framing is:

- AI = interface + orchestration
- SaaS = shared state + invariants + integrations + governance

This means time-to-first-tool is collapsing, but time-to-reliable-system is not.

## Reusable Metaphors

### Prototype vs Factory

AI gives everyone a prototype bench. SaaS still matters where the problem is running a regulated, resilient factory.

### Stage Set vs Rigging

Agents can reproduce screens, flows, and prompts. The hard part is the backstage rigging: identity, sequencing, reliability, auditability, and safety.

### Skin vs Skeleton

AI can copy the visible skin of a product. The durable value is in the skeleton and circulatory system: entity models, state transitions, and governed data flows.

## Through-Line

This packet argues that AI will absolutely eat surface-level SaaS: generic UX, shallow workflows, and thin wrappers. But it does not eliminate the need for systems that safely hold shared truth and govern action. It demotes many apps from “workspace” to “substrate.”
